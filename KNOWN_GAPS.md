# Known gaps

Things that are missing or incomplete right now, tracked here instead of in [CHANGELOG.md](CHANGELOG.md) since they're about the current state of the project rather than what changed in a given release.
Not blocking — just not built yet.

- Notes UI (creating/editing note-only entries) is built at the data layer but currently disabled in the UI pending a better interaction design.
- No tags or screenshots-per-entry.
- Trash is restore-only — no way to permanently delete a trashed folder/entry (or empty the trash) yet. Soft-deleted records just accumulate forever; there's no hard-delete anywhere in the sync protocol or the Worker's API for this to call.
- No Chrome Web Store listing — load-unpacked only.
- Switching a device's Worker URL/token doesn't clean-swap the local experience: the `Store`'s cached state (`chrome.storage.local` on the extension) isn't cleared on change, so the next sync merges (unions) local data into the new Worker rather than starting fresh against it — see `mergeState()` in `core/lib/sync.ts`.
- Still-thin automated coverage for the DOM-orchestration layer: `core/ui/*.ts` (`folders.ts`, `trash.ts`, `rail.ts`, `folderPicker.ts`, `toolbar.ts`), `extension/src/newtab/tabsPanel.ts`, `extension/src/options/main.ts`, `popup/main.ts`, `background/background.ts` have no `*.test.ts` and aren't touched by the e2e suite either — none of it has regression protection for drag-and-drop reordering, multi-select, live `chrome.tabs` sync, or the trash restore flow. `core/lib/*` (storage, sync, modal, tobyImport, linkMetadata, plus small extracted helpers like url/time/theme/backupFile/actions/favicon/uiState/config/store) is well covered (108 tests, plus 6 more for the extension's thin `chromeStore`/`chromeTabActions` adapters).
  Plain jsdom/happy-dom unit tests wouldn't actually catch what's broken most often here: every real bug found in this code during development (Playwright's simulated drag not firing real HTML5 DnD events, position-collision on restore, a timestamp-equality assumption that broke under `Date.now()`'s 1ms resolution) was a genuine browser/event-timing issue, not a logic bug behind a clean interface — this code is fundamentally about DOM/browser orchestration, not algorithms.
  A small Playwright e2e suite now exists and is CI-wired (`extension/e2e/`, run via `npm run test:e2e --workspace=extension`, its own `e2e` job in `.github/workflows/ci.yml` under `xvfb`) — but it only covers two smoke flows (folder/link creation + reload-persistence, entry multi-select's action bar + delete). The higher-risk flows still without any coverage: folder/entry drag-reorder, tabs-panel multi-select + save, live `chrome.tabs` sync, trash restore. Extending `extension/e2e/` with specs for those is the natural next step, reusing the same `fixtures.ts` (real built extension, real headed Chromium).
- No keyboard shortcuts help modal (e.g. pressing `?` to see a cheat sheet). Not worth building yet — there's currently only one real shortcut (`/` to focus search, Escape to clear it) — but revisit once there are enough shortcuts that they're not all self-evident from the UI.
- The web app (`web/`) has no drag-and-drop reordering yet — a real touch-friendly interaction-design task, deliberately deferred from the initial pass. Folders/entries can still be created, renamed, deleted, and moved between folders via buttons/modals, just not reordered by dragging.
- The web app has no PWA installability — no install manifest, no service worker, no offline-after-first-load beyond whatever the browser's own HTTP cache provides.
- The web app's manual "Add link" flow can never auto-fetch a title/favicon for the URL: `core/lib/linkMetadata.ts` fetches the target page directly from the browser, which works in the extension (bypasses CORS via `manifest.json`'s `host_permissions`) but is always blocked by CORS for arbitrary external sites from a plain web page. It falls back gracefully to asking for a title manually every time — not broken, just permanently degraded relative to the extension for this one flow. Fixing it for real would mean the Worker proxying the fetch server-side (not subject to browser CORS), which has its own considerations (the Worker would be server-side-fetching arbitrary user-supplied URLs) and wasn't in scope for this pass.
- Multi-tab write races in the web app, partially mitigated: `core/lib/storage.ts` reads/writes the entire `State` blob under one key with no per-record granularity, so two browser tabs saving within the same instant still last-write-wins with no true conflict merging. A `BroadcastChannel`-based reconciliation (`web/src/webStore.ts`) closes the _silent, indefinite_ divergence case — a tab left open picks up another tab's change almost immediately — but doesn't add real conflict resolution. Same residual risk class as the extension's analogous two-newtab-windows scenario.
- No user-visible error handling for storage-write failures throughout `core/ui/*.ts` (folders, toolbar, rail, trash, folderPicker) — every `ctx.rerender()`/`pushResource` call after a create/rename/delete/move assumes the underlying `Store.set()` succeeds. It usually does, but both backends have a real quota: `chrome.storage.local` defaults to ~10MB (no `unlimitedStorage` permission is requested), and IndexedDB's much larger best-effort browser quota can still be hit or blocked. If a write fails (quota exceeded, blocked transaction), the in-memory UI has already applied the change but it's silently never persisted — the edit appears to work and then reverts on the next reload with no error shown. The options page's and web app's own Save/Disconnect buttons got targeted fixes for this exact failure mode, but the general case (any edit inside the shared folder-browser UI) doesn't have a surfacing mechanism yet — would need a toast/banner system in `core/ui/` shared by both platforms, not a one-off fix.
