# Known gaps

Things that are missing or incomplete right now, tracked here instead of in [CHANGELOG.md](CHANGELOG.md) since they're about the current state of the project rather than what changed in a given release.
Not blocking — just not built yet.

- Notes UI (creating/editing note-only entries) is built at the data layer but currently disabled in the UI pending a better interaction design.
- No tags or screenshots-per-entry.
- Trash is restore-only — no way to permanently delete a trashed folder/entry (or empty the trash) yet. Soft-deleted records just accumulate forever; there's no hard-delete anywhere in the sync protocol or the Worker's API for this to call.
- No Chrome Web Store listing — load-unpacked only.
- Switching a device's Worker URL/token doesn't clean-swap the local experience: `chrome.storage.local`'s cached state isn't cleared on change, so the next sync merges (unions) local data into the new Worker rather than starting fresh against it — see `mergeState()` in `extension/src/lib/sync.ts`.
- No automated test coverage for the DOM-orchestration layer: `extension/src/newtab/*.ts` (`folders.ts` ~470 lines, `tabsPanel.ts` ~380), `extension/src/options/main.ts`, `popup/main.ts`, `background/background.ts` — none have a `*.test.ts`. `extension/src/lib/*` (storage, sync, modal, tobyImport, linkMetadata) is well covered (70 tests); the gap is specifically the files that wire DOM construction + event handlers + `chrome.*` calls together — drag-and-drop reordering, multi-select, live `chrome.tabs` sync, the trash restore flow.
  Plain jsdom/happy-dom unit tests wouldn't actually catch what's broken most often here: every real bug found in this code during development (Playwright's simulated drag not firing real HTML5 DnD events, position-collision on restore, a timestamp-equality assumption that broke under `Date.now()`'s 1ms resolution) was a genuine browser/event-timing issue, not a logic bug behind a clean interface — this code is fundamentally about DOM/browser orchestration, not algorithms.
  There's already a Playwright-based driver for this (`extension/.claude/skills/run-extension/driver.mjs` — launches the real built extension in a real Chromium instance), used by hand throughout development, but it isn't wired into CI (`.github/workflows/ci.yml`), so none of this ~1900 lines of glue code has any regression protection.
  Recommended approach: convert a curated set of the driver sessions already used during development into a small CI-run Playwright test suite (real assertions, headed Chromium) covering the highest-risk flows — folder/entry drag-reorder, tabs-panel multi-select + save, live tab sync, trash restore — rather than chasing unit-test coverage on files that don't unit-test well. A deeper alternative (extracting pure logic like `closestBoundary`'s index math out of the DOM-wiring so it's independently unit-testable) is worth doing eventually but is a larger refactor and wouldn't have caught the drag/DnD-specific bugs either way.
